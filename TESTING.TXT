T-Vault / TAP-DEV Security Testing Report
Date: 2026-05-01
Tested URL: http://127.0.0.1:8000/
Environment: Local Django development server on Windows, Python 3.13.5

IMPORTANT SECURITY STATEMENT
No web project can be honestly certified as "100% secure" from local testing alone.
This report records tested controls, confirmed findings, remediation applied, and residual
deployment responsibilities. Current result after hardening: LOCAL SECURITY CHECKS PASS;
production still requires real secrets, HTTPS infrastructure, monitoring, and external testing.

2026-05-01 HARDENING UPDATE
- Django upgraded from 4.2.7 to 5.2.13 LTS.
- SECRET_KEY and JWT secret are no longer hardcoded; they are environment-backed.
- DEBUG is no longer globally True; local runserver is dev mode, non-runserver defaults hardened.
- ALLOWED_HOSTS no longer uses wildcard; defaults to localhost/127.0.0.1.
- HTTPS/HSTS/secure cookie settings added for production mode.
- CSP and Permissions-Policy middleware added.
- Login, register, forgot-password, OTP verification, and mobile login rate limits added.
- Password policy strengthened to Django's standard validators with 10-character minimum.
- Registration now excludes ADMIN and requires EMAIL_VERIFY OTP activation.
- Evidence upload validation added for size, path traversal, extension, MIME type, and safe text.
- Mobile API JSON parsing now enforces content type, size limits, pagination bounds, active users,
  and generic token errors.
- IoT ingestion now validates payload size/content type and supports nonce + HMAC replay protection;
  signatures are required when DEBUG=False.
- Dependency conflicts resolved: pip check now passes.
- Security regression tests added and passing.

3. CRITICAL / HIGH FINDINGS
========================================================================

F-001 CRITICAL: DEBUG is True. STATUS: FIXED.
File: tapdev/settings.py
Evidence: DEBUG = True
Risk:
- In production, DEBUG can expose stack traces, environment details, settings, route information,
  and internal code paths.
Standards:
- OWASP Top 10 A05 Security Misconfiguration
- OWASP ASVS configuration/security logging requirements
- CSA CCM application and infrastructure security domains
Remediation applied:
- DEBUG is environment-driven.
- Local runserver can remain developer-friendly; production/check mode defaults to False.

F-002 CRITICAL: SECRET_KEY is hardcoded in source code. STATUS: FIXED.
File: tapdev/settings.py
Evidence:
- SECRET_KEY = 'tapdev-phase2-secret-key-change-in-production-xyz-2024'
Risk:
- Session signing, password reset tokens, CSRF-related signing, and cryptographic trust can be
  compromised if the repository leaks.
Standards:
- OWASP Top 10 A02 Cryptographic Failures
- CSA CCM Cryptography, Encryption and Key Management
Remediation applied:
- SECRET_KEY now loads from DJANGO_SECRET_KEY.
- Missing local secret is generated at runtime; production must set a persistent secret.

F-003 HIGH: ALLOWED_HOSTS is wildcard. STATUS: FIXED.
File: tapdev/settings.py
Evidence: ALLOWED_HOSTS = ['*']
Risk:
- Host header attacks, cache poisoning, poisoned password reset links, and routing ambiguity.
Standards:
- OWASP Top 10 A05 Security Misconfiguration
Remediation applied:
- ALLOWED_HOSTS now loads from DJANGO_ALLOWED_HOSTS and defaults to localhost/127.0.0.1.

F-004 HIGH: HTTPS hardening is missing. STATUS: FIXED IN SETTINGS.
Command: python manage.py check --deploy
Evidence:
- SECURE_HSTS_SECONDS not set.
- SECURE_SSL_REDIRECT not True.
- SESSION_COOKIE_SECURE not True.
- CSRF_COOKIE_SECURE not True.
Risk:
- Session and CSRF tokens can be exposed over non-HTTPS transport.
Standards:
- OWASP ASVS transport security and session management controls
- OWASP Top 10 A02 Cryptographic Failures
- CSA CCM Infrastructure and Virtualization Security / Application Security
Remediation applied:
- SECURE_SSL_REDIRECT, SESSION_COOKIE_SECURE, CSRF_COOKIE_SECURE, HSTS, HSTS preload,
  SameSite, HttpOnly session cookie, nosniff, referrer policy and X-Frame-Options configured.
Deployment note:
- Real production must provide HTTPS/TLS at the proxy or app boundary.

F-005 HIGH: Django version is outdated and behind known security releases. STATUS: FIXED.
Files:
- requirements.txt pins Django==4.2.7
- runtime shows Django 4.2.7
OSINT:
- Django official security releases in 2025 include patched 4.2.x releases for SQL injection,
  directory traversal, denial of service, and other issues.
Risk:
- Known vulnerabilities may affect this application depending on database backend and features used.
Standards:
- OWASP Top 10 A06 Vulnerable and Outdated Components
- CSA CCM Vulnerability Management
- CISA KEV principle: vulnerability intelligence must inform prioritization
Remediation applied:
- requirements.txt now specifies Django>=5.2.13,<5.3.
- Installed runtime upgraded to Django 5.2.13.

F-006 HIGH: CSRF-exempt API endpoints need stronger compensating controls. STATUS: PARTIALLY FIXED.
Files:
- apps/mobile_api/views.py
- apps/iot_gateway/views.py
Evidence:
- @csrf_exempt appears on mobile login and several API endpoints.
- IoT device_push is token-authenticated with X-Device-Token.
Risk:
- CSRF exemption can be valid for token APIs, but endpoints must enforce strict method,
  authentication, replay resistance, rate limiting, payload size limits, content type checks,
  and audit logging.
Current concerns:
- mobile_login has no rate limiting.
- JWT secret derives from TAPDEV_CONFIG default and should be environment-backed.
- IoT device token appears static and bearer-like; no rotation/replay nonce observed.
Standards:
- OWASP ASVS authentication, session, API and input validation controls
- OWASP Top 10 A07 Identification and Authentication Failures
Remediation applied:
- Mobile login rate limiting added.
- Mobile JSON content type and payload size validation added.
- Mobile token errors are generic.
- JWT secret now environment-backed.
- IoT payload size/content-type validation added.
- IoT nonce replay protection and HMAC verification added; HMAC is required in production mode.
Remaining:
- Add persistent API auth failure logging and token rotation UI.

4. MEDIUM FINDINGS
========================================================================

F-007 MEDIUM: No brute-force lockout or rate limiting confirmed for login/OTP. STATUS: FIXED.
Evidence:
- django-ratelimit is installed in the environment, but no decorator or middleware was found on
  login, forgot password, or verify OTP views.
Risk:
- Credential stuffing and OTP brute force.
Remediation applied:
- Rate limits added for login, registration, forgot-password, OTP verification, and mobile login.

F-008 MEDIUM: Password policy is weak. STATUS: FIXED.
File: tapdev/settings.py
Evidence:
- Only MinimumLengthValidator with min_length 6 is enabled.
Risk:
- Weak passwords are accepted.
Remediation applied:
- UserAttributeSimilarityValidator, CommonPasswordValidator, NumericPasswordValidator enabled.
- Minimum length increased to 10.

F-009 MEDIUM: requirements.txt is incomplete compared with configured apps. STATUS: IMPROVED.
File: requirements.txt
Evidence:
- requirements.txt contains only Django, Pillow, python-magic, django-crispy-forms.
- Project imports/use settings for rest_framework and channels; runtime has many packages installed
  globally, but project requirements do not reproduce the environment.
Risk:
- Builds are not reproducible; production may fail or run unintended package versions.
Remediation applied:
- requirements.txt updated with Django 5.2, DRF, Channels, PyJWT, ReportLab, django-celery-beat.
Remaining:
- A full minimal lockfile should still be produced before deployment.

F-010 MEDIUM: Dependency consistency problems exist in the active environment. STATUS: FIXED.
Command: python -m pip check
Evidence:
- pandas-ta requires pandas >= 2.3.2 but pandas 2.3.1 is installed.
- torchaudio requires torch == 2.9.1 but torch 2.10.0 is installed.
Risk:
- Runtime instability in ML/AI-related features.
Remediation applied:
- pip check now reports no broken requirements.

F-011 MEDIUM: No automated test suite currently exists. STATUS: FIXED PARTIALLY.
Command: python manage.py test
Evidence:
- Found 0 tests. NO TESTS RAN.
Risk:
- Security fixes can regress silently.
Remediation applied:
- 5 security regression tests added.
Remaining:
- Expand tests for object-level permissions, evidence chain integrity, IoT HMAC, and OTP lockout.

F-012 MEDIUM: JSON embedded with |safe should be reviewed.
Evidence:
- Several templates render JSON into JavaScript using |safe.
Risk:
- Safe rendering is acceptable only if data is produced by json.dumps/json_script style escaping.
Required fix:
- Prefer Django json_script or verified json.dumps with strict escaping for chart data.

1. Move SECRET_KEY and JWT secrets to environment variables.
2. Set DEBUG=False for production.
3. Replace ALLOWED_HOSTS=['*'] with explicit domains.
4. Upgrade Django to current supported security release.

2. Add failed-login logging and lockout/backoff.

Priority 2 - API hardening:
1. Keep CSRF exemption only for true token APIs.
2. Add replay protection for IoT device pushes.
3. Enforce max body sizes and content types.
4. Rotate device tokens and JWT secrets.

Priority 3 - Testing:
1. Add Django tests for auth, OTP, registration role restriction, route redirects, custom 404.
2. Add API tests for missing/invalid Bearer token and invalid device token.
3. Add evidence upload tests for file size/type/path traversal.
4. Add chain integrity tests for reorder/delete/duplicate/fake timestamp scenarios.
